# 소켓/네트워크 학습 키워드 정리 (CSAPP 11장 중심)

## BSD 소켓

- BSD 소켓은 커널이 제공하는 네트워크 I/O 인터페이스 집합이다.  
- 핵심 아이디어: `연결-바인딩-대기-수락-송수신-종료`를 수행하는 일련의 API를 통해 애플리케이션이 네트워크 스택을 직접 다루지 않고 통신한다.
- CSAPP 11장에서 자주 보이는 흐름:
  - 클라이언트: `socket()` → `getaddrinfo()`(주소 후보 구성) → `connect()`
  - 서버: `socket()` → `bind()` → `listen()` → `accept()`
- 파일 디스크립터(fd)를 반환한다는 점에서, 소켓은 TCP/UDP를 위한 `open`과 유사한 인터페이스이다.
- 포트 기반 접속을 위해 `sockaddr`(주소 구조)와 결합해 바인딩/연결을 수행한다.
- CSAPP 코드 스타일에서 보듯이 실제 프로젝트는 시스템 호출을 그대로 쓰기보다 `open_clientfd`, `open_listenfd` 같은 **래퍼 함수**를 사용해 반복 패턴을 감춘다.
- 동작 개념상 “API 계약”이고, 내부 동작은 프로토콜 스택(TCP/IP)에 위임된다.

## IP

- IP는 네트워크 계층 계층에서 **주소 지정, 라우팅, 단편화/재조립의 기본 토대**를 제공한다.
- “IP 데이터그램”은 보통 **IP 헤더 + 상위 프로토콜 페이로드**로 구성된다.  
  즉, 페이로드 내부에는 TCP/UDP 같은 상위 프로토콜의 데이터가 들어간다.
- CSAPP 11장 맥락에서 IP는 보통 “호스트 찾기/전송 길항점으로 전달”의 기반을 의미한다.
- 포트는 IP가 아니라 TCP/UDP 계층의 값으로, “어느 프로세스(서비스)”로 가는지 지정한다.
- IPv4/IPv6가 있을 때 주소 구조의 차이가 생기며, 이때 `getaddrinfo()`가 주소 후보를 `sockaddr_in`/`sockaddr_in6` 기반으로 구성한다.
- 라우팅에서 목적지 IP 경로를 못 찾으면 연결/전송 시도는 실패(`EHOSTUNREACH`, `ENETUNREACH` 등 형태)로 나타날 수 있다.

## TCP

- TCP는 연결 지향(커넥티드) 바이트 스트림 프로토콜이다.
- CSAPP가 강조하듯 TCP의 핵심 역할:
  - 신뢰성(재전송, ACK 기반 확인)
  - 순서 보장(시퀀스 번호)
  - 흐름 제어(수신자 버퍼/윈도우 기반)
  - 혼잡 제어(네트워크 상태 반영 전송량 제어)
- 연결/해제 과정:
  - 연결 수립: 3-way handshake (`SYN`, `SYN-ACK`, `ACK`)
  - 종료: FIN/ACK 교환(상태 전이 및 TIME-WAIT 구간 포함)
- CSAPP 관점에서 서버는 `listen` 소켓으로 수신을 대기시키고 `accept` 시점마다 별도 연결 소켓에 대한 상태를 가진다.
- `send()`/`write()` 호출은 “한 번에 보낸 크기만큼” 전송된다고 보장되지 않는다. TCP는 바이트 스트림이라 데이터 경계가 보존되지 않으며, 애플리케이션은 메시지 경계를 직접 다뤄야 한다.
- `SOCK_STREAM` 소켓이 바로 TCP 기반 연결형 통신의 기본이다.

## HTTP

- HTTP는 애플리케이션 계층 프로토콜이다. TCP 위에서 동작하며 요청/응답(Req-Resp) 의미론을 가진다.
- CSAPP 예제에서처럼 `GET /path HTTP/1.0 ...` 같은 텍스트 라인을 파싱해서 리소스 경로를 추출하고, 응답 헤더와 본문을 구성한다.
- “한 개 연결 = 한 요청/응답” 패턴은 HTTP/1.0/간단 구현에 가깝고, 실제 구현은 keep-alive(연결 재사용) 유무에 따라 다르다.
- HTTP 특성상 “메시지 경계”는 텍스트 포맷(라인/빈 줄 구분, 헤더/바디 규칙)로 구현되므로 TCP 바이트 스트림과 다르게 파싱 계층이 필요하다.
- 프록시/학습용 웹서버 구현에서는:
  - 요청 라인 파싱
  - 정적 파일 요청인지, 프록시용 URI인지 판별
  - 응답 생성(`HTTP/1.0` 상태줄, `Content-Length`, MIME 또는 기본 타입 추론)
  순서로 동작한다.

## file descriptor

- fd는 정수형 핸들(테이블 인덱스)로, 프로세스마다 fd 테이블이 있고, fd 값은 현재 프로세스 공간의 인덱스만 나타낼 뿐 실제 소켓/파일 객체는 커널이 관리한다.
- 같은 테이블에서 열린 파일, 소켓, 파이프를 구분해 핸들링한다.
- 장점:
  - `read`, `write`, `close` 같은 동일 I/O 인터페이스를 파일/소켓에 공통 적용
  - 타입별 분기는 커널/런타임 계층에서 처리
- CSAPP 관점에서 소켓 API가 “소켓을 fd로 반환”하는 이유가 여기서 드러난다.
- 따라서 애플리케이션은 fd를 통해 동작을 수행하고, 실제 객체의 수명/버퍼/상태(연결 상태, 대기열 등)는 커널 쪽 자료구조가 가진다.

## DNS

- DNS는 호스트명 → IP 주소(그리고 필요하면 포트/서비스 번호 매핑)에 대한 분산 디렉터리 시스템이다.
- CSAPP 11장에서 `getaddrinfo()`는 이름 해석 + 주소 후보 집합 생성의 표준 진입점으로 자주 등장한다.
- 주요 동작:
  - `hostname`과 `service`를 받아 주소 후보 `addrinfo` 리스트(ai_next로 연결) 생성
  - 클라이언트는 이 후보를 순서대로 시도하며 `socket()`/`connect()`로 접속
  - IPv4/IPv6 혼합 환경에서 우선순위/폴더미팅(운영체제 정책)에 따라 후보를 선택
- 실패 시 즉시 실패 처리보다 “모든 후보 시도 후 최종 에러 반환” 패턴이 흔하다.
- 캐시/TTL의 개념은 DNS 질의 성능/부하에 큰 영향을 준다(실전에서는 resolver/OS 캐시, /etc/hosts 등이 영향).

## 개념 연결 정리 (정리용)

- `BSD 소켓 API`는 TCP/HTTP 웹서버 구현의 기본 틀이다.
- `IP`는 어디로 보낼지(호스트), `TCP`는 어떻게 보낼지(연결/신뢰성), `HTTP`는 무엇을 말할지(요청/응답 형식), `DNS`는 호스트 이름을 주소로 바꾸는 시작점이다.
- `file descriptor`는 이 모든 I/O 자원을 같은 방법으로 다루게 해주는 운영체제의 통합 추상화다.
